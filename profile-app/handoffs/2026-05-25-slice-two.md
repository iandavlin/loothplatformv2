## What surprised me (the 5-liner)

1. **Backfilled profiles have no `lat`/`lng`** â€” only `location_text` (raw
   xprofile strings). The location-radius filter in the directory therefore
   excludes ALL 660 backfilled users. With the filter on, you see only the
   handful who've explicitly saved via Places autocomplete (today: just Ian).
   With the filter off, all 663 appear with text-only location. The slice-3-
   ish fix is a geocoding pass over `users` where `location_text IS NOT NULL
   AND lat IS NULL` â€” should be a single `bin/geocode.php` script.
2. **The polymorphic credentials shape (`owner_type` + `owner_id`) works
   without a single per-table join in slice 2,** but the schema endpoint
   has to lie a bit â€” it documents credentials as a top-level user thing
   (which it is for `owner_type='profile'`), with no mention of the
   `'practice'` branch that exists in the column constraint. Slice 3 will
   need to either add a `credential_owner` discriminator at the schema
   level or live with the gap.
3. **Curated catalogs hold up well for instruments/scenes/skills**, but
   credentials is the squishy one: the 60-entry seed missed several
   common cards (Galloup-affiliated programs are over-represented; PRS
   Private Stock authorized service is missing). The free-text fallback
   saved this â€” first new user just types it in â€” but the resulting
   `profile_credentials` row has `catalog_id=NULL`, which means it
   doesn't filter cleanly in directory faceted search. Worth adding a
   nightly job that proposes catalog rows from popular free-text values
   for Ian to approve.
4. **Native HTML5 typeahead pattern is fine for credentials,** but it
   silently fails when the typed query is < 2 chars â€” debouncing skips
   the fetch entirely. I kept the minimum (server returns up to 50 hits
   per request, which is fine for 60-row catalogs) but a real production
   directory would also want a "show all" affordance for users who
   want to browse rather than search.
5. **Postgres `earthdistance` returns miles by default** (via the
   `point(lng,lat) <@> point(lng,lat)` operator). I sanity-checked
   against a known city pair and miles match. Worth knowing because
   slice-3 practices will want the same logic. Also installed `cube` as
   a dependency â€” both required `CREATE EXTENSION` as superuser, which
   the `profile-app` role doesn't have. Documented in `bin/`-flavored
   instructions for future migrations.
