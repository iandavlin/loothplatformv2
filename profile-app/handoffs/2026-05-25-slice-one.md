## What slice one ships

A working **front-end editor at `/profile/edit`** (dev only) covering **About,
Location, Socials**. End-to-end loop verified: log into WP → JWT mints into
`looth_id` cookie → /profile/edit lazy-creates a `profiles` row → edit fields →
save via `/profile-api/v0/me/*` → reload persists → public viewer sees the
filtered subset.

Auth is **RS256 JWT in a `looth_id` cookie**, minted by WP (mu-plugin), verified
by profile-app. This is the auth precedent for every future Looth service —
copy it elsewhere, do not re-implement.

## The auth contract (READ THIS)

```
sub        = UUIDv5(LOOTH_IDENTITY_NAMESPACE, lower(trim(email)))
algorithm  = RS256
issuer     = https://dev.loothgroup.com
ttl        = 30 days
cookie     = looth_id; Domain=.dev.loothgroup.com; Path=/; HttpOnly; Secure; SameSite=Lax
alt input  = Authorization: Bearer <token>   (for future native app)
```

- Minting lives ONLY in `profile-auth.mu-plugin.php`. Every other service is a
  pure verifier.
- profile-app does not query WP. It only opens `/etc/looth/jwt-public.pem` and
  calls `JWT::decode()` (firebase/php-jwt). No DB lookup for auth itself; the
  user-row fetch is a single SELECT by `uuid`.
- Refresh: on 401, `edit.js` calls `/wp-json/looth/auth/refresh` (which sees
  the WP session cookie) to re-mint, then retries the original request once.

## Viewer-role + per-precision location

`Profile::renderForViewer($full, $role)` is the single source of truth — used by
both `/me/preview` (editor toggle) and `/user/<uuid>` (public read). The editor's
JS toggle mirrors it client-side for instant feedback but the server is authoritative.

Visibility:
- `me` → sees all sections (their own profile)
- `friend` → sees public + members + (later) friend-only; for now resolves like `member`
- `member` → sees public + members
- `public` → public only

Location precision is per-grant on the users row (defaults: public='city',
members='city', friends='address'). `Profile::renderLocation()` coarsens
text/components AND lat/lng for non-'address' grants. `'hidden'` strips entirely.

## What surprised me (the 5-liner)

1. **Keypair perms: spec said `root:www-data 640`, reality needs `root:looth-dev 640`.**
   On this box `www-data` is only the nginx + FPM master user; the WP mu-plugin
   actually runs as the pool user `looth-dev`. A literal read of the spec would
   have left the mu-plugin unable to open `/etc/looth/jwt-private.pem`. Same
   pattern will apply on live (`looth-live`). The public key (644 root:root)
   is the only safe-to-copy half.
2. **Cookie scope: spec said `.loothgroup.com` but dev needs `.dev.loothgroup.com`.**
   If the dev mu-plugin set the bare apex domain, a future live mu-plugin
   minting under `.loothgroup.com` would shadow it (browsers send the more
   specific cookie first, but the same-named broader cookie overwrites on the
   next mint). Slice one's mu-plugin scopes the cookie to the host the JWT
   was issued for; the live cutover swaps both the `iss` claim AND the
   `Domain=` attribute together.
3. **Refresh-flow has a silent failure mode.** `edit.js` retries once via
   `/wp-json/looth/auth/refresh` on a 401. That endpoint's `permission_callback`
   is `is_user_logged_in()`, so if the WP session itself also expired, the
   refresh returns 401 and the editor shows "signed out" with no actionable
   prompt. Acceptable for slice one; slice two should redirect to the WP
   login interstitial instead.
4. **UUIDv5 namespace is duplicated in two source trees now.** profile-app/config.php
   has `LOOTH_IDENTITY_NAMESPACE`; the mu-plugin has `LOOTH_AUTH_NAMESPACE`,
   same value. Plus: WP email changes silently desync the JWT `sub` from
   `users.uuid` (uuid is frozen from first-seen email; mint always recomputes).
   Both want the same slice-two fix: shared `/opt/looth/identity.php` package
   + `looth_uuid` cached in WP usermeta on first mint.
5. **Places API key was already in `wp_options.wpgmza_google_maps_api_key`,**
   and the slice-zero `SELECT ON looth_dev.*` grant let profile-app read it
   directly per request — no config-file plumbing needed. Plus an audit
   finding from the JWT smoke tests: **115 of 1809 WP users have empty
   `user_email`**, so their JWT `sub` is `UUIDv5(ns, "")` which resolves to
   nothing in profile-app and they 401 from `/profile/edit`. Pre-existing
   ghost accounts; slice-zero already skipped them, slice one just makes
   the lockout visible.
