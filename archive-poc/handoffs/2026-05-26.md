# archive-poc — Session Handoff (2026-05-26)

## What this handoff is for

**Goal of the next session:** Replace the WP-admin "Archive POC Config" dash with a **front-end inline editor** on `loothgroup.com/archive-poc/`, modeled on the lg-layout-v2 FE editor pattern. Authors edit rows in place, see live changes, save via REST.

This handoff captures (a) why we're doing it, (b) the existing patterns we'll borrow from lg-layout-v2, (c) the proposed architecture for archive-poc's version, and (d) staged work + open decisions.

---

## Why

The current WP-admin dash (`ArchivePocDash.php`) renders all rows as a flat table with the same field set: ID, Title, Type, Layout, Column, Audience, Tag, Query (JSON). But different row types need different fields:

| Type | Actually needs |
|---|---|
| `activity-strip` | nothing (auto-runs, server-injected) |
| `cta-bar`, `sponsors`, `local-looths` | just title (data lives in other sections) |
| `static` rail/grid | tag, kind, sort, limit, etc. |
| `events-upcoming` | limit |
| `tag-random` | tag pool, limit |
| `video-promo` | side, video_id, html, aspect (totally different shape) |

User feedback: **"This is really hard to parse."** And: **"I'm not seeing how I select a new row to be a column."** (The Column field is sidebar placement, not row layout — confusing.)

The right fix is per-type forms with visual context. Easier still: edit them where they render.

## Existing pattern to borrow from: lg-layout-v2 FE editor

Files of interest under `/home/ubuntu/projects/lg-layout-v2/`:

- `src/FeEditor.php` — boots the editor: enqueues assets, adds `lg-can-edit` body class, gates on `current_user_can('edit_post')` + `?lg_edit=1`, exposes a header "Edit" button.
- `src/EditorRest.php` — REST endpoints under `/wp-json/lg-layout-v2/v1/blocks/`:
  - `POST /blocks/update` — persist block prop changes
  - `POST /blocks/insert` — add a new block
  - `POST /blocks/delete` — remove
  - `POST /blocks/move` — reorder
  - All gated by nonce + capability check (`edit_post`)
- `src/EditorPickers.php` — typed pickers (image, rich-text). Manifest's `editor.custom_picker` declares which picker a block uses; this class dispatches.
- `assets/lg-fe-editor.js` — client side:
  - Walks `<lg-edit>` markers the server-side Renderer emitted (one per block)
  - Each marker pairs with its host element
  - Builds a per-block pill bar from the block's manifest `editor.pill_buttons`
  - Wires `contenteditable` on the block's `inline_editable_props`
  - Dispatches custom pickers when triggered
  - Every mutation hits `/wp-json/lg-layout-v2/v1/blocks/{op}`
  - On successful save → `location.reload()` so server-rendered HTML stays canonical (no client-side render)
- `assets/lg-fe-editor.css` — pill bar, spinner, edit-mode chrome.

**Key design choices that matter:**

1. **Server-rendered is the source of truth.** Save → reload. No client-side rendering to maintain.
2. **Per-block manifests describe what's editable.** A block declares its editable props + pill buttons in its manifest. Adding a new editable field = manifest change, no JS code.
3. **Pickers are pluggable.** Image picker reuses WP media library. Custom pickers register in `EditorPickers.php`.
4. **Spinner is refcounted** so concurrent saves don't flicker.

## Proposed architecture for archive-poc FE editor

### Activation
- URL: `https://loothgroup.com/archive-poc/?ap_edit=1` (different param than `lg_edit` to avoid conflicts)
- Gate: viewer must be a WP admin (check via `wordpress_logged_in_*` cookie + REST nonce lookup; archive-poc has no WP context so verify via a REST call to a new `/wp-json/looth/v1/whoami` field returning capability info)
- Or simpler gate: check the `lg_tier` cookie + a new `lg_edit_capable=1` cookie minted by a mu-plugin when user has `manage_options`. archive-poc PHP reads the cookie, conditionally emits editor markers.

### Server-side markers (in archive-poc/web/index.php + _render-main-row.php)
For each rendered row, emit:
```html
<section class="row row--rail" data-row-id="most-liked" data-ap-edit="1" data-ap-row-config='<json>'>
  ...row content...
</section>
```
The `data-ap-row-config` attribute carries the current row's config (id, title, type, layout, column, audience, query). JS reads it, builds the edit panel.

### Client-side editor (new file: `archive-poc/web/ap-fe-editor.js`)
Mirrors lg-fe-editor.js structure:
- Walk all `[data-ap-edit="1"]` rows
- Add a floating pill bar on hover: ☰ drag handle · ✎ edit · ⬆ ⬇ reorder · ✕ delete
- "Edit" opens a side panel with **type-specific fields** (the missing piece in current dash). Field schemas declared in JSON, similar to lg-layout-v2 manifests.
- Drag-reorder via Sortable.js or native HTML5 DnD
- Save: POST to `/archive-api/v0/_config` (existing endpoint) with the full updated config. Reload on success.

### Type-specific field schemas
Keep this as static JSON in the JS for now — small set of row types, schema rarely changes. Example:
```js
const ROW_SCHEMAS = {
  'static': {
    label: 'Rail / Grid of posts',
    fields: ['id', 'title', 'layout', 'column', 'audience', 'tag', 'kind', 'sort', 'limit', 'min_likes', 'max_age_days'],
  },
  'tag-random': {
    label: 'Random tag rail',
    fields: ['id', 'title', 'column', 'audience', 'limit', 'candidate_pool'],
  },
  'video-promo': {
    label: 'Two-column promo (video + HTML)',
    fields: ['id', 'title', 'column', 'audience', 'side', 'video_id', 'html', 'aspect'],
  },
  // ...
};
```
Each field has a type (text, number, select, json, rich-html). The panel renders ONLY the relevant fields for the row's type.

### Backend save endpoint
Use the existing `POST /archive-api/v0/_config` — it already accepts the full config JSON and writes to `config.json` atomically. No new backend needed.

### Authoring the video-promo HTML
Use either:
- A `<textarea>` with a small "preview" button → easier
- Or integrate `Quill` / `TipTap` for actual rich editing → nicer but adds 50-100KB JS
Recommendation: textarea + preview on the first pass, upgrade later.

## Open design decisions for the next session

1. **Activation gate** — cookie-based (`lg_edit_capable`) vs query-param + REST nonce check at first render? Cookie is faster but requires a mu-plugin to mint; nonce is one REST call per page-load but cleaner.
2. **Field-schema location** — hard-coded in JS, or fetched from `/archive-api/v0/_schema` for dynamic config? Hardcoded is simpler; fetched lets non-engineers update schemas later.
3. **Drag-reorder library** — pull in Sortable.js (~30KB minified) or roll native HTML5 DnD? Sortable handles edge cases (touch, keyboard a11y). Native is zero-dep but more code.
4. **Atomic save vs partial save** — every field change saves immediately, or accumulate then "Save" button? lg-layout-v2 does per-change autosave. archive-poc rows are smaller; could either way.
5. **Lock the dash UI behind a feature flag** — keep WP-admin dash working as a fallback while FE editor matures? Yes, recommended. Add a `define('LG_ARCHIVE_POC_FE_EDITOR', true)` toggle.
6. **What about the OTHER dash sections** (sponsors, local_looths, cta_member, cta_public)? These don't render inline on the page, so FE editing doesn't naturally fit. Keep them in WP-admin as Phase 2, OR add a floating "Site-wide config" panel reachable from the FE editor toolbar.
