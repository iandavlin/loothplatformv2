# BB-mirror — Session Handoff (2026-05-27, step 7 done)

## What this project is

Read-side strangler for BuddyBoss/bbPress forum threads. Forum URLs render
out of bb-mirror's own SQLite + FPM pool at native-static speed; writes
still round-trip through BB REST so notifications/moderation/mentions/
presence keep working unchanged.

## Fixes that landed during deploy

1. **Sync URL** in [deploy/bb-mirror-sync.php](deploy/bb-mirror-sync.php) was
   `/forums/_sync`; corrected to `/bb-mirror-api/v0/_sync` to match nginx.
2. **Int cast on `$p->post_author`** in [api/v0/_sync.php](api/v0/_sync.php) —
   WP_Post returns it as string, our typed function rejected it.
3. **FK-off during backfill** in [bin/backfill.php](bin/backfill.php) plus an
   orphan-prune pass at commit. ON DELETE CASCADE in the schema only
   triggers if you delete *through* SQLite; backfill walks WP's posts table
   which has orphan topics/replies from manual deletes. Pruning the dangling
   rows post-insert is the cheapest fix.
4. **`umask(0002)` in [config.php](config.php)** — SQLite creates WAL/SHM
   files with the process umask, and the default 022 made them group-r-only.
   Two processes (bb-mirror pool + looth-dev pool via loothdevs group)
   write to the same DB; both need group-write. 0002 + group `loothdevs`
   ownership handles it.

## SQLite ownership cheat sheet (do not chown without thinking)

```
index.sqlite{,-wal,-shm}   bb-mirror:loothdevs   664
```

- `bb-mirror` user: web FPM pool that renders pages
- `www-data` user (looth-dev pool that runs `_sync.php`): in `loothdevs` group
- `umask(0002)` in [config.php](config.php) keeps new WAL files at 664

Querying as `ubuntu` with `sqlite3` will fail in WAL mode (SQLite requires
write on -shm even for read). Use `sudo -u bb-mirror sqlite3 ...` or add
`ubuntu` to `loothdevs` group.

## How to test (post-deploy)

```bash
# Counts
sudo -u bb-mirror sqlite3 /home/ubuntu/projects/bb-mirror/index.sqlite \
  "SELECT 'forums',COUNT(*) FROM forum UNION ALL
   SELECT 'topics',COUNT(*) FROM topic UNION ALL
   SELECT 'replies',COUNT(*) FROM reply;"

# Hit the sync endpoint directly (simulates the mu-plugin POST)
curl -sk -X POST https://127.0.0.1/bb-mirror-api/v0/_sync \
  -H "Host: dev.loothgroup.com" -H "X-BB-Mirror-Sync: 1" \
  -H "Content-Type: application/json" \
  -d '{"kind":"topic","id":<topic_id>,"action":"upsert"}'

# Trigger a real hook via wp-cli (non-blocking; check log after ~1s)
cd /var/www/dev && sudo -u www-data wp eval 'do_action("bbp_edit_topic", <id>);'

# Check the WP debug log for any sync errors
sudo tail -f /var/www/dev/wp-content/debug.log | grep bb-mirror

# Re-backfill (idempotent — uses INSERT OR REPLACE)
cd /var/www/dev && sudo -u www-data wp eval-file \
  /home/ubuntu/projects/bb-mirror/bin/backfill.php
```
