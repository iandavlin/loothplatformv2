1. Two services, one database

Slim app — user-facing API

Synchronous HTTP. The browser talks to this. Repo: lg-stripe-billing (this one).

  • POST /v1/checkout — create Stripe Checkout session
  • POST /v1/portal — create Customer Portal session
  • GET /v1/return — Stripe redirect handler, immediate grant
  • GET /v1/tiers — frontend tier listing
  • GET /health

No webhooks. No cron. Reads/writes lg_membership on user-driven paths only.

WP plugin — async event sync

WP cron. Background polling. Repo: lg-member-sync (separate, not yet built).

  • Polls Stripe Events API (cursor-based)
  • Polls Patreon OAuth (replaces existing patreon plugin)
  • Writes to lg_membership (customers, subscriptions, entitlements)
  • Arbitrates roles via lg_role_sources
  • Sole writer of wp_usermeta.wp_capabilities
  • Sends welcome email on first provision
user → Slim → Stripe (checkout / portal) user-facing flow WP cron → Stripe Events API → arbiter → wp_caps async catch-up WP cron → Patreon OAuth → arbiter → wp_caps async catch-up

The two services share one MySQL instance. Slim writes to lg_membership on the synchronous return-URL path; WP plugin writes to it on the polling path. lg_membership.entitlements is the canonical source of truth for "what can this customer do."

5. Flow walkthroughs

A. Checkout — synchronous user path

  1. Frontend POST /v1/checkout with price_id email country.
  2. CheckoutService resolves regional price via ProductRepository.resolvePriceForCountry.
  3. CheckoutService calls StripeGateway.createCheckoutSession, returns clientSecret.
  4. User completes payment in Stripe's embedded form. Stripe redirects to /v1/return?session_id=....
  5. ReturnHandler retrieves the session from Stripe, finds tier from price.
  6. CustomerManager.findOrCreate → row in customers.
  7. SubscriptionRepository.upsert → row in subscriptions.
  8. EntitlementManager.grantMembershipFromSubscription → row in entitlements.
  9. Slim returns redirect to home / "you're a member" page.
  10. No role write here. The WP plugin's next poll tick (≤ 5 min) sees the new entitlement and updates wp_capabilities.

Tables touched: customers · subscriptions · entitlements · audit_log

B. WP poller tick (every ~5 min)

  1. WP cron fires lg_member_sync_tick.
  2. Read cursor from lg_event_cursor. Call GET /v1/events?starting_after=<cursor> on Stripe API.
  3. For each event:
    • checkout.session.completed → noop if Slim already provisioned (row exists), otherwise upsert
    • customer.subscription.updated → upsert sub row, recompute entitlement
    • customer.subscription.deleted → mark sub canceled, revoke entitlement
    • invoice.paid → upsert sub (status, periods)
    • invoice.payment_failed → log, no role change
    • charge.refunded → revoke entitlement, audit log (handled manually for now)
  4. Update cursor to last processed event ID.
  5. For every customer with a state change in this tick: report to arbiter.
  6. Arbiter recomputes wp_capabilities based on all sources' opinions, writes wp_usermeta.
  7. Same tick polls Patreon OAuth and reports those opinions to the same arbiter.

Tables touched: subscriptions · entitlements · lg_role_sources · wp_usermeta · lg_event_cursor · audit_log

C. Cancellation

  1. User clicks "cancel" in Customer Portal. Stripe records cancel_at_period_end=true, fires no immediate role change.
  2. Period ends. Stripe records the subscription as deleted.
  3. Next poller tick picks up the customer.subscription.deleted event.
  4. EntitlementManager.revokeForSubscription → entitlements.revoked_at = NOW().
  5. Arbiter sees Stripe source now reports tier=null; if Patreon also reports null, writes looth1 (lapsed). Preserves admin / bbp roles.

D. Refund — manual for now

  1. User emails support requesting refund.
  2. Admin issues refund in Stripe Dashboard.
  3. Stripe records charge.refunded event.
  4. Next poller tick will eventually consume it. Until automated handling lands, admin can manually revoke entitlement via SQL or admin UI.

Automated handling deferred — explicit choice.
