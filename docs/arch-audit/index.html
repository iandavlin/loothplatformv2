How a user is served — ground-truth audit

dev.loothgroup.com · derived only from nginx, FPM pools, app code & live DBs (no docs/handoffs) · 2026-06-02

The site is a strangler: one nginx front door behind a cookie gate fans every URL out to one of seven PHP-FPM pools, each its own app with its own database. WordPress is now just one tenant among several — most user-facing surfaces are standalone apps that bypass WP entirely.

Database topology & sprawl

⚠ Split-brain on the MariaDB layer

WordPress core serves from looth_import (the live clone). But every satellite that reads WP’s MariaDB directly — events, profile-app’s WP-user bridge, and lg-stripe-billing — is still pointed at looth_dev, the older fixture DB. They are reading a different, staler WordPress than the one WordPress itself serves.

Two more dupes: the Postgres looth.discovery schema is a half-finished SQLite→PG migration (started by a chat that ran out of context) — its article_blobs (299) is live and serves CPT pages, while its content_item (1,830) duplicates the SQLite search index; both run in parallel. And member identity is mirrored across three stores — SQLite person, PG forums.person, and PG profile_app.users.
